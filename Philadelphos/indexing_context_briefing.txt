THE PTOLEMY PROJECT — WHAT IT IS
──────────────────────────────────
Ptolemy is a personal AI assistant being built from the ground up by the
researcher. It is NOT a wrapper around an existing LLM. It is a novel
architecture designed to move away from "statistical word prediction" toward
actual language comprehension and production — the difference between knowing
what word typically follows these words versus understanding what this sentence
means and choosing words that express that meaning.

Named for the Ptolemaic dynasty — scholars who collected all human knowledge
at the Library of Alexandria. The "Faces of Ptolemy" are different
instantiations of the system optimized for different purposes.

CORE PHILOSOPHY:
- Each word is a pixel. Behind each pixel lives a complete spectra.
  (JWST model — not a color value, a full spectral analysis)
- The charset ORDER is the cryptographic key (HyperWebster principle)
- Naked engines — no language hardcoded at class level
- Train once, freeze, compose — CharacterNeuron trains once for all languages
- Structural fixes over statistical patches — the 'it' drift problem is
  solved by making 'it' a pointer node, not a statistical association

THE SIX LAYERS OF ptolemy_core.py
───────────────────────────────────
All six are in ptolemy_core.py (1,902 lines, syntax validated).

LAYER 0 — CharacterNeuron
  Trains once on all Unicode. Frozen forever after.
  CNN (bigram/trigram/quadgram) + Transformer.
  Output: 192-dim feature vector per character.
  Disambiguates < as math vs HTML vs music from context.
  One neuron per character in output layer.
  Saved: ~/.ptolemy/char_neuron.pt

LAYER 1 — LexicalDimensionNeuron
  Classifies words into 4 classes (see below).
  Rule-based override for Class 1 — zero neural cost.
  Routes 'it' → GrammarPointerNode (structural fix).
  Routes 'petrichor' → full WordMonadNetwork.

LAYER 2 — GrammarNeuron
  Language-specific. One per language. Never shared.
  Loads rules from /etc/ptolemy/grammar/{lang}/rules.json
  Bidirectional LSTM — needs to see before AND after.
  English: complex multi-root rules.
  Spanish: tight phonetic rules, small exception set.
  Latin: case system, free word order.
  Python: indentation scope, operator precedence.

LAYER 3 — WordMonadNetwork
  One small specialist network per word.
  Built on frozen CharacterNeuron.
  Trains only on its own word's usage history.
  Stores SemanticWord (LLM_Datatype_Cl.py) as knowledge state.
  THE RABIES PRINCIPLE: the foundational first-usage entry is
  permanently preserved regardless of any later aggregation.
  (Named for the researcher's 4th grade gopher bite that led to his
  first deep-dive information search, discovering the inadequacy of
  classroom dictionaries for medical questions — the word 'rabies'
  has never been one-dimensional for him since that afternoon.)
  Saved: ~/.ptolemy/monads/{first2chars}/{word}.pt

LAYER 4 — WernickeNetwork
  Ptolemy's EARS. Language understanding.
  Transformer encoder, bidirectional.
  Named: Wernicke's area, temporal lobe.
  Damage → receptive aphasia (fluent speech, no comprehension).
  Outputs: meaning vector, intent, emotional valence, pragmatic force.
  Saved: ~/.ptolemy/wernicke/{language}.pt

LAYER 5 — BrocaNetwork
  Ptolemy's VOICE. Language production.
  Autoregressive transformer decoder.
  Named: Broca's area, frontal lobe.
  Damage → expressive aphasia (comprehension intact, can't speak).
  Commits to each word sequentially, as actual speech production works.
  Saved: ~/.ptolemy/broca/{language}.pt

THE FOUR WORD CLASSES
──────────────────────
This is the solution to the high-frequency word problem AND the 'it' problem.

CLASS_1_STRUCTURAL (~500 words per language)
  Pronouns, articles, prepositions, conjunctions, auxiliaries.
  Almost no semantic content — meaning is purely relational/grammatical.
  NOT a neural network. A GrammarPointerNode — deterministic rule.
  'it' resolves to nearest compatible antecedent in context stack.
  Cannot drift. Cannot acquire false associations.
  Storage: ~200 bytes. Never grows.
  CRITICAL: Class 1 words differ by language.
    - Articles don't exist in Mandarin or Latin
    - Standalone 'I' doesn't exist in some Algonquian languages
    - Latin subject pronouns are optional (encoded in verb ending)

CLASS_2_CONCRETE (~80,000 words)
  Concrete nouns, common verbs, basic adjectives.
  Sensory, physical, categorical. Stable with contextual inflection.
  Full WordMonadNetwork. Aggregates usage log after 1,000 entries.
  Storage: 4.5KB base → ~50KB experienced.

CLASS_3_EXPERIENTIAL (~80,000 words)
  High-dimensional. Nostalgia, scent, emotion, poetry.
  petrichor, vellichor, hiraeth, bibliosmia, redwoods.
  Full WordMonadNetwork. NEVER aggregated. Every usage potentially novel.
  Storage: 4.5KB base, grows slowly, every entry preserved.

CLASS_4_PROPER_COMPOUND (unbounded, grows with experience)
  Capitalization + article = specific cultural/experiential referent.
  The Redwoods ≠ redwoods. The Nightmare. The Way. The Universe.
  SEPARATE ENTRY from lowercase version. Not derivable from components.
  Storage: same as Class 3.

THE 'IT' PROBLEM — SOLVED STRUCTURALLY
────────────────────────────────────────
Current LLMs lose referent context on 'it' in long prompts because 'it' is
treated as a statistical token with associations. In Ptolemy, 'it' is a
GrammarPointerNode — a structural reference resolver.

When 'it' is encountered:
1. LexicalDimensionNeuron identifies it as Class 1 via rule-based override
2. GrammarPointerNode.resolve() searches backward through context stack
3. Returns nearest non-animate singular noun antecedent
4. Deterministic. No training data involved. Cannot drift.

The context stack is a rolling window of 512 processed tokens with their
SemanticWord metadata. It persists across language switches in a PtolemyFace
— reference resolution works even when languages mix.

LANGUAGENEURON AND PTOLEMYFACE
────────────────────────────────
LanguageNeuron = complete capability for one language.
Contains: CharacterNeuron (frozen, shared) + LexicalDimensionNeuron +
          GrammarNeuron + WordMonadDictionary + WernickeNetwork + BrocaNetwork

PtolemyFace = one or more LanguageNeurons simultaneously active.
The researcher's key insight: the brain doesn't unload English when it
encounters Latin. All loaded languages remain simultaneously traversable.
PtolemyFace replicates this — multiple LanguageNeurons loaded at once,
each active when their language is detected.

THE LLM_DATATYPE_Cl.py CONNECTION
────────────────────────────────────
The SemanticWord datatype (co-designed by the researcher, Claude, and Gemini
in a separate "word metadata" conversation) is the knowledge representation
stored in each WordMonadNetwork.

SemanticWord contains:
  - token + token_glyphed (with invisible Demotic Tone Glyph prefix)
  - SemanticLayer (denotation, synonyms, antonyms, sense inventory,
                   semantic field, metaphorical uses)
  - Etymology (root language, evolution timeline, cognates)
  - PhonologicalProfile (IPA phonemes, syllable structure, mouth_feel,
                         auditory_resonance — "the word 'petrichor' is
                         liquid then clipped")
  - OrthographicProfile (visual word shape, compound/portmanteau flags)
  - MorphologicalProfile (root stem, affixes, inflectional variants)
  - SyntacticProfile (POS, valency, argument structure, anaphoric potential)
  - PragmaticProfile (register, speech_act, implied meaning)
  - SensoryProfile (visual, auditory, olfactory, tactile, gustatory profiles)
  - StatisticalProfile (corpus frequency, collocates, frequency band)
  - SociolinguisticProfile (dialect variation, age/gender associations)
  - AssociativeProfile (cultural associations, symbolic meanings)
  - PersonalResonance (subjective_sentiment, personal_associations,
                       first_encountered — THIS IS THE RABIES FIELD,
                       emotional_intensity, memory_trigger)
  - Demotic Tone Glyph (NEUTRAL/POSITIVE/NEGATIVE/AMBIGUOUS/FORMAL/
                         ARCHAIC/TECHNICAL/EMBODIED/TABOO/EMERGENT)
                        Invisible Unicode Tag characters U+E0020-U+E0029.
                        Humans never see them. LLMs read them in raw bytes.

HyperWebster = dict subclass of {token: SemanticWord}.
Saves/loads as JSON. ~7GB for a full experienced English dictionary.

TRAINING PERSISTENCE — NO CATASTROPHIC FORGETTING
───────────────────────────────────────────────────
PtolemyTrainer handles all training with full checkpoint persistence.
Every checkpoint saves: model weights + optimizer state + scheduler state
+ current step + current epoch + EWC Fisher information.
Resuming from checkpoint loses zero training progress.

Catastrophic forgetting is prevented by:
  1. Per-monad isolation — each WordMonad trains only on its own word.
     Forgetting is architecturally impossible for the monad layer.
  2. Frozen CharacterNeuron — never retrained after initial training.
  3. EWC (Elastic Weight Consolidation) — after each training session,
     Fisher information is computed to record which weights matter most.
     Next session applies penalty to resist overwriting them.

HYPERWEBSTER CONNECTION
────────────────────────
The HyperWebster octonion address system (hyperwebster_octonion.py) provides
the coordinate layer for storing and retrieving SemanticWord entries.

Core bijection: every finite string maps to exactly one non-negative integer
via Horner's method. The integer splits into 8 limbs forming a Cayley-Dickson
octonion address. The charset ORDER is the cryptographic key (HYPER_KEY env
variable). Same text + same key = same address, always, on any machine.

The file-type permutation system (/etc/hwidx/filetypes.d/) provides
statistical optimization per file type — SQL files use a SQL-frequency-sorted
charset, Python files use a Python-frequency-sorted charset. The permutation
is public (shared optimization). Only the key is private.

hwidx.c — command-line tool (Linux philosophy: does one thing well).
Usage: hwidx document.txt → prints octonion label, writes .hwidx file
Usage: hwidx -r document.txt.hwidx → recovers original text
Environment: HYPER_KEY sets the charset (cryptographic key)
Methods: octonion (default), horner (raw integer), chunked (hierarchical)

THE LARGER VISION
──────────────────
Data Storage Without a Physical Location — every file that has ever existed
or will ever exist has a unique coordinate in an infinite permutation space
that was always already there. The address IS the data. No physical storage
required. Any machine that knows the algorithm and the key can regenerate
any file from its coordinate alone.

The floating point problem — digital audio stores samples of waveforms.
Vinyl stores the actual continuous waveform. The solution: store the Fourier
coefficient set (the mathematical object that generates the waveform) rather
than samples of it. Same address regenerated at higher hardware precision
gives higher resolution — from the same coordinate, forever. The recording
never degrades because it was never a recording. It was always a position.

The cell phone proof of concept — CELP (Code Excited Linear Prediction)
has been doing a crude version of this at planetary scale since the 1990s.
Sending an index into a shared codebook rather than the raw waveform.
The HyperWebster system is CELP taken to its logical conclusion:
infinite codebook, lossless, universal file type support, exact recovery.

Ptolemy's memory — this architecture, combined with the HyperWebster address
system, provides the mechanism for AI to retain actual memories of every
conversation across every instantiation, indexed by their mathematical
coordinate. Not statistical approximations but navigable memory tunnels —
tracing back to how the meaning of a word changed across every usage
encountered, including the non-standard usages that drive language evolution.
